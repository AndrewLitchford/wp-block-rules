Hard Rules

WordPress Custom Block Architecture

This document defines how all custom WordPress blocks are built, organized, edited, and maintained across all projects. These rules are not guidelines, they are the standard. Deviations require explicit justification and sign-off.

01

Block Technology Standard

1.1

ACF Blocks Are the Default

All custom structured blocks are built as ACF blocks using ACF Pro's block registration API. Native Gutenberg block registration is not used for custom blocks unless there is a specific technical requirement that ACF blocks cannot satisfy.

1.2

Scope of ACF Blocks

ACF blocks are used for all structured block types regardless of whether the data lives outside the block or not. A block does not need to be tied to a CPT, taxonomy, or queryable dataset to justify being an ACF block. Simple one-instance page-level blocks (hero, CTA band, stats bar, testimonial, etc.) are ACF blocks. The decision to use ACF is about the editing architecture, not the data architecture.

1.3

ACF Blocks for CPT-Driven Blocks

Any block that queries or displays data from a Custom Post Type, taxonomy, or external data source is always an ACF block. No exceptions.

1.4

GenerateBlocks Is a Utility Plugin, Not a Block-Building Tool

GenerateBlocks is installed and available for one-off layout needs that fall outside the structured block library (e.g., a developer-assembled promo section for a single page). It is never used as a substitute for building a proper ACF block. If a layout pattern appears more than once or originates from a Figma design, it becomes an ACF block.

1.5

Core Blocks for Free-Form Content

WordPress core blocks (Paragraph, Heading, List, Image, Quote, etc.) are used for long-form editorial content areas — blog posts, resource pages, documentation — where editors need genuine free-form composition. They are not used to assemble structured design components.

02

Canvas and Editing Philosophy

2.1

The Canvas Is Always a Preview

The editor canvas is a pixel-accurate preview of the rendered block. It is never an editing surface. Editors do not click into the canvas to type, change images, or modify any content. All editing happens in the sidebar.

2.2

No Inline Canvas Editing

RichText components, inline toolbar controls, and any other form of on-canvas interaction are not used in custom blocks. The canvas renders a static PHP template via ACF's mode: 'preview' setting.

2.3

All Fields Live in the Sidebar

Every editable element of a block — text, media, toggles, layout settings, style overrides — is a field in the sidebar Inspector panel. Editors have one place to look. Always.

2.4

ACF Block Mode

All ACF blocks are registered with 'mode' => 'preview'. Blocks never use 'mode' => 'edit' (which surfaces ACF fields overlaid on the canvas) or 'mode' => 'auto'.

acf_register_block_type(array(
    'name'            => 'hero',
    'title'           => 'Hero',
    'render_template' => 'blocks/hero/hero.php',
    'mode'            => 'preview',
    'supports'        => array( ... ),
));
03

Sidebar Organization — The Three-Tab Structure

3.1

All Block Sidebars Use Three Tabs

Every ACF block's field group is organized into exactly three ACF Tab fields:

  • Content — all text fields, media fields, CTA fields, repeaters. Everything an editor touches on every use.
  • Style — color overrides, typography overrides, spacing overrides. These deviate from the design system default and are escape hatches, not primary controls.
  • Advanced — layout mode, responsive behavior toggles, custom CSS class, anchor/ID, any developer-facing settings.
3.2

Content Tab Comes First

The Content tab is always the first tab in the field group. Editors land there by default.

3.3

Style Tab Is for Overrides Only

The block's default appearance should be correct with zero Style fields set. Style controls exist to handle intentional, approved deviations from the design system on a per-instance basis. If a style setting is always required to make the block look right, it belongs in CSS, not in a Style field.

3.4

Advanced Tab Is Low-Traffic

Settings in the Advanced tab are configured once at build time or rarely touched by editors. If a developer finds themselves putting things in the Advanced tab that editors will adjust frequently, those fields should be reconsidered for the Content or Style tab.

04

ACF Field Type Rules

4.1

Short Text with No Formatting

Use the Text field type for: eyebrow labels, button labels, phone numbers, addresses, short single-line data. No formatting controls are available and none are needed.

4.2

Text That May Need Inline Formatting

Use the WYSIWYG field type for: headings (a single word may need to be bolded or italicized), subheadings, intro copy, any field where an editor might need to highlight a word, add an inline link, or apply basic emphasis.

4.3

WYSIWYG Fields Use a Minimal Toolbar

All WYSIWYG fields use a custom minimal toolbar unless a specific field explicitly requires more. The minimal toolbar contains only: Bold, Italic, Link, Remove Formatting. Media upload is disabled on all WYSIWYG fields unless explicitly required.

// Register the minimal toolbar globally
add_filter('acf/fields/wysiwyg/toolbars', function($toolbars) {
    $toolbars['Minimal'] = array();
    $toolbars['Minimal'][1] = array('bold', 'italic', 'link', 'removeformat');
    return $toolbars;
});

// Apply to any WYSIWYG field that needs it
array(
    'type'         => 'wysiwyg',
    'toolbar'      => 'Minimal',
    'media_upload' => 0,
)
4.4

Long Body Copy

Use WYSIWYG with the basic toolbar for multi-paragraph body copy where editors may need ordered/unordered lists in addition to bold, italic, and links.

4.5

Media Fields

Use ACF's Image field type for images (returns image array, ID, or URL per field setting). Use URL or Text for video embed URLs. Never use the WordPress media uploader outside of ACF's native image/file field types in custom blocks.

4.6

Repeater Fields

Use ACF Repeater fields for lists of items (team members, stats, testimonials, feature items). Repeaters always live in the Content tab. Minimum and maximum row counts are set to match the design constraints — if the Figma design shows 3–6 stat items, set min: 3, max: 6.

4.7

Conditional Logic

ACF's native conditional logic is used to show and hide fields based on other field values. A field should never be visible in the sidebar if it has no effect given the current state of other fields. Example: overlay opacity and overlay color fields are hidden when the media type is set to None.

05

Block Variants, Variations, and Separate Blocks

5.1

The Decision Framework

Use the following test when a Figma design shows multiple versions of a block:

Difference TypeSolution
Same fields, different visual presentation onlyACF Block Style (is-style-*) + CSS
Same fields, different default values or layout modeBlock Variation (inserter preset with default attributes)
1–2 additional conditional fieldsAttribute toggle + ACF conditional logic
Fundamentally different field schemaSeparate ACF block
5.2

The Structural Test for Separate Blocks

Ask: if an editor opens this block expecting to work on Variant A but finds themselves in Variant B's configuration, would 4 or more fields be irrelevant or confusing to them? If yes, it is a separate block. If the conditional logic can cleanly hide those fields, it stays as one block with toggles.

5.3

Block Variations Are Inserter Presets

registerBlockVariation is used to give editors distinct named entries in the block inserter for common configurations of a block. Each variation pre-populates attributes to match a Figma variant. Editors see "Hero – Full Bleed" and "Hero – Compact" as separate inserter items, but they are the same underlying block. After insertion, a Layout control in the Advanced tab lets them switch if needed.

5.4

Block Styles Are for Visual-Only Differences

registerBlockStyle is used when the only difference between two design variants is CSS — dark vs. light overlay, outlined vs. filled CTA, etc. Block styles add a modifier class (is-style-dark) to the block wrapper. PHP templates and ACF field schemas are identical between styles. All style differences are handled in CSS.

5.5

Block Patterns for Page-Level Presets

For each major Figma-designed page template or section composition, register a Block Pattern that pre-populates a full set of blocks with placeholder content. This is what editors start from when building a new page, not a blank inserter. Patterns are registered in the theme and organized by category (e.g., yourtheme/heroes, yourtheme/cta-sections).

06

Spacing and Padding Rules

6.1

Default Block Padding

Every block's default padding is defined in the block's stylesheet and requires no field values to be set. The block looks correct at these defaults out of the box.

64pxTop
64pxBottom
64pxLeft / Right
6.2

Padding Override Fields

Every block includes padding override fields in the Style tab. These use ACF's Number or Text fields and cover top, bottom, left, and right padding independently. These fields are blank by default. When set, they override the CSS default via an inline style or a scoped CSS custom property on the block wrapper.

6.3

Margin Override Fields

Every block includes optional top and bottom margin override fields in the Style tab. Default block margin is 0. Margins between sections are controlled by the block below using its top padding, not by the block above using its bottom margin, except where a specific design requires otherwise.

6.4

Responsive Spacing

Padding and margin override fields accept a single value applied across all breakpoints unless responsive control is explicitly required by the design. When responsive control is needed, three separate sub-fields are used (Desktop, Tablet, Mobile) rather than a single responsive value field.

6.5

No Hardcoded Spacing in Templates

PHP block templates never contain hardcoded padding or margin inline styles. All spacing comes from CSS defaults or from ACF field values applied to a style attribute on the wrapper element via PHP.

07

Design Token and Theme Integration

7.1

Figma Tokens Live in theme.json

All design tokens from Figma (colors, spacing scale, type scale, radii) are defined as custom properties in theme.json under settings.custom. They are referenced in block stylesheets as var(--wp--custom--*). No raw hex values, pixel values, or font names are hardcoded in block CSS.

7.2

Block Supports for Global Styles Integration

All ACF blocks declare appropriate supports in their registration to integrate with WordPress Global Styles where applicable. At minimum:

'supports' => array(
    'anchor'  => true,
    'align'   => array('wide', 'full'),
    'spacing' => array(
        'padding' => true,
        'margin'  => true,
    ),
),

Color and typography supports are added on a per-block basis where the design system permits editor-level overrides. They are not added globally to all blocks by default.

7.3

CSS Architecture Per Block

Each ACF block has its own scoped stylesheet (blocks/block-name/block-name.css). Styles do not bleed between blocks. The block wrapper always carries a unique class (wp-block-yourtheme-block-name) and all CSS is scoped to that class.

08

PHP Template Rules

8.1

One Template Per Block

Each block has one PHP render template. There are no shared templates between blocks. If two blocks share significant markup, that markup is extracted to a partial (blocks/partials/) and included in each block's template separately.

8.2

All Field Output Is Escaped

Every ACF field value output in a PHP template is escaped appropriately:

  • esc_html() for plain text
  • wp_kses_post() for WYSIWYG output
  • esc_url() for URLs
  • esc_attr() for HTML attribute values
8.3

Conditional Rendering

PHP templates use conditional checks before rendering any element that depends on a field having a value. Empty fields never produce empty HTML wrappers in the rendered output.

<?php if ( get_field('subheading') ) : ?>
    <p class="hero__subheading"><?php the_field('subheading'); ?></p>
<?php endif; ?>
8.4

Block Wrapper Uses get_block_wrapper_attributes()

The outermost wrapper element in every PHP template uses get_block_wrapper_attributes() to ensure WordPress-generated classes (including is-style-*, alignment classes, and anchor IDs) are applied correctly.

<section <?php echo get_block_wrapper_attributes(['class' => 'hero']); ?>>
09

Block File Structure

9.0

Standard Directory Layout

Every ACF block lives in blocks/block-name/ and contains the following files. Field groups are registered in code (fields.php), not saved to the database via the ACF UI. Field groups are never managed through the WordPress admin and exported to PHP — they are authored in PHP from the start.

blocks/
  hero/
    hero.php          # Render template
    hero.css          # Block-scoped styles
    hero.js           # Block-specific JS (if needed)
    block.json        # Block metadata (title, icon, keywords, category)
    fields.php        # ACF field group registration for this block
10

Naming and Registration Conventions

10.1

Block Namespace

All custom blocks are registered under the theme or project namespace: yourtheme/block-name. The namespace is consistent across all projects unless a plugin requires its own namespace.

10.2

Block Names Are Kebab-Case

Block names use kebab-case: hero, cta-band, team-grid, location-hero. No underscores, no camelCase.

10.3

ACF Field Keys Are Prefixed and Consistent

ACF field keys follow the pattern field_{blockname}_{fieldname}: e.g., field_hero_heading, field_hero_media_type. This prevents key collisions across blocks and makes field references readable.

10.4

Block Categories

All custom blocks are registered under a custom block category (yourtheme-blocks) so they appear grouped and separate from core blocks in the inserter.

11

What Never Goes in a Custom Block

11.0

Hard Prohibitions

  • Free-form layout assembly. If an editor needs to freely arrange elements, that is a GenerateBlocks or core blocks use case, not a custom block.
  • Functionality that belongs in a plugin. Forms, ecommerce, maps, and similar features are handled by dedicated plugins. ACF blocks can wrap or display plugin output, but they do not replicate plugin functionality.
  • Hardcoded content. No custom block contains hardcoded copy, images, or data that should be editable. If it is visible on the page, it comes from a field.
  • Page layout/template logic. Custom blocks are section-level components. Full page layout (header, footer, sidebar, column structure) is handled at the theme template level, not in blocks.
12

Summary Decision Tree

12.0

When Starting a New Block from a Figma Design

  1. Is it a structured, designed component?
    → ACF block.
  2. Does it query a CPT or external data?
    → ACF block with dynamic query logic in the template.
  3. Does the Figma file show multiple variants?
    • Only visual/CSS differences → One block + Block Styles.
    • Same fields, different defaults → One block + Block Variations as inserter presets.
    • 1–2 conditional extra fields → One block + ACF conditional logic.
    • Different field schema → Separate block.
  4. Does it need free-form rich text composition?
    → Core blocks inside an ACF InnerBlocks field or a dedicated core block area.
  5. Is it a one-off layout not worth formalizing?
    → GenerateBlocks. Document why.
